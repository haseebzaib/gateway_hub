{# ═══ SECTION 1 — Topic Structure ═══ #}

Topic Structure

Every topic follows this pattern. Your server subscribes to these — the gateway only publishes, never subscribes.

metacrust_<mac>/{source}/{device_id}/{data_type}

This gateway's prefix: loading…  —  derived from the eth0 MAC address. Each physical gateway has a unique prefix so topics never collide across a fleet.

gateway_id

Uniquely identifies this physical gateway across your entire fleet. Automatically derived from the eth0 MAC address at startup — you never configure this manually. Format: metacrust_ followed by the 12-character MAC without colons (e.g. metacrust_b827eb1a2c3d). Every gateway produces a different prefix, so two gateways with identically-named devices never collide on the broker.

{source}

The interface protocol — always one of these exact strings:

rs232 RS-232 serial rs485 RS-485 / Modbus RTU modbus_tcp Modbus TCP
{device_id}

The name you typed when adding the device in the Interfaces page. It is not auto-generated — whatever name you entered is exactly what appears in the topic. To find or change it, go to Interfaces → RS232 / RS485 / Modbus TCP and check the Device ID field.

{data_type}

Three fixed values:

sensor_data — live readings from the device analytics   — computed stats, trends & alert states events     — status changes & alert events (on-change only)

Example — DustTrak particle counter, RS232, named dustrak_01, on gateway metacrust_b827eb1a2c3d

sensor_datametacrust_b827eb1a2c3d/rs232/dustrak_01/sensor_data
analyticsmetacrust_b827eb1a2c3d/rs232/dustrak_01/analytics
eventsmetacrust_b827eb1a2c3d/rs232/dustrak_01/events

RS485 temperature + humidity sensor named env_b2, same gateway

sensor_datametacrust_b827eb1a2c3d/rs485/env_b2/sensor_data
analyticsmetacrust_b827eb1a2c3d/rs485/env_b2/analytics
eventsmetacrust_b827eb1a2c3d/rs485/env_b2/events
{# ── Gateway ID note ── #}
🔑 Every message payload includes gateway_id as the first field. For MQTT this is redundant — the gateway ID is already in the topic prefix — but it is included so payload-only consumers do not need to parse the topic. For any other transport in the future the payload is self-contained.
Your server should index on gateway_id + source + device_id together to uniquely identify a data stream. Two gateways can have devices with the same name; the gateway_id makes them distinct.
{# ═══ SECTION 2 — sensor_data ═══ #}

Topic 1  …/sensor_data

Published every interval (configured above). Contains all current metric readings from the device. The metric names inside readings match exactly what the device reports.

// DustTrak particle counter — RS232, device_id = "dustrak_01"
// Metrics: PM1.0, PM2.5, PM10 in µg/m³
{
  "gateway_id": "metacrust_b827eb1a2c3d",  // ← identifies which gateway sent this
  "ts":         1714567890123,             // Unix timestamp in milliseconds (UTC)
  "device_id":  "dustrak_01",             // as configured in Interfaces → RS232
  "source":     "rs232",
  "status":     "ok",                     // "ok" | "warning" | "error" | "offline"
  "readings": {
    "pm1_0": { "value": 8.1,  "unit": "µg/m³", "quality": "good" },
    "pm2_5": { "value": 12.3, "unit": "µg/m³", "quality": "good" },
    "pm10":  { "value": 28.7, "unit": "µg/m³", "quality": "good" }
  }
}

// RS485 multi-sensor — device_id = "env_b2"
// Metrics: temperature, humidity, CO2
{
  "gateway_id": "metacrust_b827eb1a2c3d",
  "ts":         1714567890123,
  "device_id":  "env_b2",
  "source":     "rs485",
  "status":     "ok",
  "readings": {
    "temperature": { "value": 24.5, "unit": "°C",  "quality": "good" },
    "humidity":    { "value": 65.2, "unit": "%",   "quality": "good" },
    "co2":         { "value": 420,  "unit": "ppm", "quality": "stale" }
  }
}

// Modbus TCP flow meter — device_id = "flow_meter_01"
{
  "gateway_id": "metacrust_b827eb1a2c3d",
  "ts":         1714567890123,
  "device_id":  "flow_meter_01",
  "source":     "modbus_tcp",
  "status":     "ok",
  "readings": {
    "flow_rate":    { "value": 3.42, "unit": "m³/h", "quality": "good" },
    "total_volume": { "value": 1250, "unit": "m³",   "quality": "good" }
  }
}

Field reference

gateway_idMAC-derived identifier for this gateway (metacrust_ + 12 hex chars). Unique per physical device. Use this to distinguish data from multiple gateways posting to the same endpoint.
tsTimestamp in milliseconds since Unix epoch (UTC). Parse with new Date(ts) in JS or datetime.fromtimestamp(ts/1000) in Python.
statusOverall device health: ok = all good, warning = degraded but reading, error = device fault, offline = no communication.
readings.*.qualitygood = fresh valid reading, stale = device not responding but last value retained, error = read failed.
{# ═══ SECTION 3 — analytics ═══ #}

Topic 2  …/analytics

Published every interval alongside sensor_data. Contains computed analytics for the device. Sections are dynamic — only sections that have data are included in the message. A newly connected device will have no stats or trends for the first ~5 minutes while data accumulates.

ℹ stats and trends appear after ~5 minutes of continuous readings. active_alerts appears only when you have configured alert rules in the Insights page. If a device has no rules and insufficient data, the message will only contain the header fields. 
// Full example — all sections present (env_b2, rs485)
{
  "gateway_id": "metacrust_b827eb1a2c3d",
  "ts":         1714567890123,
  "device_id":  "env_b2",
  "source":     "rs485",

  // Rolling window statistics — averages, min/max, std deviation
  "stats": {
    "temperature": {
      "5min": { "avg": 24.1, "min": 23.8, "max": 24.5, "stddev": 0.22 },
      "1hr":  { "avg": 23.9, "min": 22.1, "max": 25.1, "stddev": 0.80 },
      "24hr": { "avg": 23.5, "min": 21.0, "max": 26.3, "stddev": 1.10 }
    },
    "humidity": {
      "5min": { "avg": 65.0, "min": 64.5, "max": 65.8, "stddev": 0.30 },
      "1hr":  { "avg": 64.2, "min": 62.0, "max": 66.5, "stddev": 0.90 },
      "24hr": { "avg": 63.8, "min": 58.0, "max": 70.2, "stddev": 2.10 }
    }
  },

  // Trend detection — slope and direction over the last ~5 minutes
  "trends": {
    "temperature": {
      "direction":     "rising",   // "rising" | "falling" | "stable"
      "slope_per_min": 0.05,       // units per minute (positive = rising)
      "ttt_minutes":   45          // minutes to nearest alert threshold; omitted if no rule
    },
    "humidity": {
      "direction":     "stable",
      "slope_per_min": 0.01
    }
  },

  // Alert rule states — one entry per configured rule
  "active_alerts": [
    {
      "rule_id":      "ab12cd34",
      "metric":       "temperature",
      "condition":    "gt",        // "gt" | "lt" | "gte" | "lte" | "eq"
      "threshold":    30,
      "severity":     "warning",   // "warning" | "critical"
      "state":        "ok",        // "ok" = below threshold, "fired" = threshold exceeded
      "triggered_at": null         // ms timestamp when fired; null if state = "ok"
    }
  ]
}
{# ═══ SECTION 4 — events ═══ #}

Topic 3  …/events

Published immediately on change — not on a timer. Your server receives this the moment a device goes offline, recovers, or an alert rule fires or resolves. Use this topic for real-time alerting. Both event types share the same topic; distinguish them by the type field.

// ── Event type 1: status_change ──────────────────────────────────────────
// Fired when device health transitions between ok / warning / error / offline
{
  "gateway_id": "metacrust_b827eb1a2c3d",
  "ts":         1714567890123,
  "device_id":  "dustrak_01",
  "source":     "rs232",
  "type":       "status_change",
  "payload": {
    "previous": "ok",             // status before the change
    "current":  "error",          // "ok" | "warning" | "error" | "offline"
    "message":  "No response from device after 3 attempts"
  }
}

// ── Event type 2: alert ───────────────────────────────────────────────────
// Fired when an alert rule is triggered or when it clears back to normal
{
  "gateway_id": "metacrust_b827eb1a2c3d",
  "ts":         1714567900000,
  "device_id":  "env_b2",
  "source":     "rs485",
  "type":       "alert",
  "payload": {
    "rule_id":   "ab12cd34",
    "metric":    "temperature",
    "condition": "gt 30",         // human-readable condition string
    "value":     31.2,            // current reading that triggered the rule
    "threshold": 30,
    "severity":  "warning",       // "warning" | "critical"
    "state":     "fired"          // "fired" = threshold exceeded, "resolved" = back to normal
  }
}

{# HTTPS endpoint reference panel #}
{# ── URL construction ── #}

How URLs Are Constructed

The gateway combines the host, port, and path you enter above into a full HTTPS URL. Port 443 is omitted from the URL (standard HTTPS default).

https://{host}:{port}/{path}
port 443https://api.example.com/ingest
custom porthttps://api.example.com:8443/ingest
{# ── Gateway identification callout ── #}
🔑

How your server tells gateways apart

Unlike MQTT — where the topic prefix metacrust_b827eb1a2c3d/… identifies the gateway — HTTPS POSTs go to a plain URL with no built-in identity. Every POST body therefore includes gateway_id as the first field. Your server must read this field to distinguish which physical gateway sent the data. Two gateways can have devices with identical names (e.g. both have a device called env_b2); without gateway_id the data would be indistinguishable.

Index your database on gateway_id + source + device_id together. That triple is globally unique.

{# ── Request format ── #}

Request Format

All three endpoints receive HTTP POST requests with a JSON body. Your server must return HTTP 2xx — any other response is logged as a failure and the gateway retries on the next interval.

POST /ingest HTTP/1.1
Host: api.example.com
Content-Type: application/json
Authorization: Bearer <token>          ← only if Bearer auth is configured
X-API-Key: <key>                       ← only if API Key auth is configured
Content-Length: <n>

{ "gateway_id": "metacrust_b827eb1a2c3d", ... }  ← gateway_id always first
ℹ Basic Auth uses a standard Authorization: Basic <base64(user:pass)> header. The gateway only sends application/json — no cookies, multipart, or form data.
