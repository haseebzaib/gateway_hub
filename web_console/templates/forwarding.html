{# Topics reference panel #}
{# ═══ Topic structure ═══ #}

Published Topics

The gateway only publishes — it never subscribes. Topics are fixed; your server subscribes to metacrust/# or to specific branches below.

This gateway's ID: loading… — derived from its network MAC address, unique per physical unit, never configured by hand.

availabilitymetacrust/<gateway>/status
sensor datametacrust/<gateway>/sensors/{type}/{id}
device healthmetacrust/<gateway>/device/metrics
anomaly eventsmetacrust/<gateway>/device/anomalies

{type} is the interface (rs232, rs485, modbus_tcp) and {id} is the channel/device name from the Interfaces page — e.g. …/sensors/rs232/ch0.

📶 The status topic is retained and carries online / offline via MQTT Last-Will — your dashboard knows within seconds when a gateway drops, without polling.
{# ═══ Message envelope ═══ #}

Message Envelope

Every message on every topic is one JSON envelope with the same five fields. Only data varies by kind. Messages publish the moment data arrives (device health at the interval you chose in “What to send”). QoS is configurable above; delivery survives uplink outages — unsent data is buffered on the gateway and replayed in order on reconnect.

// kind = "sensor_data" — topic …/sensors/rs232/ch0
{
  "gateway_id":   "mcx-…",
  "kind":         "sensor_data",
  "source":       { "type": "rs232", "id": "ch0" },
  "timestamp_ms": 1784046190644,
  "data":         { /* the sensor payload exactly as the device runtime produced it,
                       e.g. samples: [{ "name": "pm25", "value": 12.5, "unit": "mg/m3" }] */ }
}

// kind = "device_metrics" — topic …/device/metrics
{ "gateway_id": "…", "kind": "device_metrics",
  "source": { "type": "device", "id": "metrics" }, "timestamp_ms": …,
  "data": { "cpu": {…}, "memory": {…}, "temperature_c": 44.8,
            "filesystem": {…}, "emmc": {…}, "uptime_sec": 4512 } }

// kind = "anomaly_event" — topic …/device/anomalies
{ "gateway_id": "…", "kind": "anomaly_event",
  "source": { "type": "device", "id": "cpu.temp" }, "timestamp_ms": …,
  "data": { "severity": "Warning", "category": "Crossed a safe limit",
            "metric_label": "CPU Temperature", "value": 78.0,
            "headline": "CPU Temperature reached 78 °C, past the warning limit 75 °C.",
            "message": "…what could be wrong, in plain English…" } }

Field reference

gateway_idUnique per physical gateway. Index your database on gateway_id + source — that pair is globally unique across a fleet.
kindsensor_data | device_metrics | anomaly_event — how to route/parse data.
timestamp_msMilliseconds since Unix epoch, UTC. new Date(ts) in JS, datetime.fromtimestamp(ts/1000) in Python.

{# ── URL construction ── #}

How the URL Is Built

https://{host}:{port}{ingest path}

Default ports (443/80) are omitted. Everything you selected in “What to send” arrives at this one endpoint — the kind field inside each envelope tells your server what it is.

{# ── REST format ── #}

REST Format — the body is always a JSON array

Usually one envelope per POST. After an uplink outage the gateway replays its backlog in batches — same array shape, just more items — so your handler is identical either way. Reply HTTP 2xx to acknowledge; anything else and the gateway keeps the data buffered and retries.

POST /ingest HTTP/1.1
Content-Type: application/json
X-Gateway-Id: mcx-…              ← always present
Authorization: Bearer <token>       ← per the Authentication setting
X-API-Key: <key>                    ← (API Key mode)

[
  { "gateway_id": "…", "kind": "sensor_data",
    "source": { "type": "rs232", "id": "ch0" },
    "timestamp_ms": 1784046190644,
    "data": { "samples": [{ "name": "pm25", "value": 12.5, "unit": "mg/m3" }] } },
  { "gateway_id": "…", "kind": "anomaly_event", "source": { "type": "device", "id": "cpu.temp" },
    "timestamp_ms": …, "data": { "severity": "Warning", "headline": "…", "message": "…" } }
]
🔑 Envelope fields are identical to MQTT (gateway_id, kind, source, timestamp_ms, data). Index on gateway_id + source.
{# ── GraphQL format ── #}

GraphQL Format

Selectable under Payload Format. The gateway POSTs your mutation with one variable: $events — the same array of envelopes as REST. Edit the mutation text in the form to match your schema; a response containing an errors array counts as a failure and the batch is retried.

POST /graphql HTTP/1.1
Content-Type: application/json

{
  "query": "mutation Ingest($events: [JSON!]!) { ingestGatewayData(events: $events) }",
  "variables": { "events": [ { "gateway_id": "…", "kind": "sensor_data", … } ] }
}
