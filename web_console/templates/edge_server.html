Route Configuration

Define the HTTP paths and MQTT topics that devices are allowed to send data to.

HTTP / HTTPS Endpoints
Protocol selects the listener. HTTPS uses TLS and requires the HTTPS listener to be enabled; mTLS can additionally verify device certificates. Method POST records new telemetry/events. PUT is for devices that update one current state object. Path is one URL route for this endpoint. Add another endpoint later when another path is needed. Payload JSON is parsed into fields, raw is stored as bytes/text, form reads key-value fields. Device ID source chooses where the gateway gets the device identity: payload field, header, path segment, or fixed value. Pipeline stores first, runs anomaly detection, then forwards only when a Data Forwarding profile is selected.
Configure Data Forwarding profiles

No HTTP endpoints

Add a path that HTTP devices can call on the gateway.

MQTT / MQTTS Topics
Protocol MQTT is plain broker traffic. MQTTS uses TLS and requires the MQTTS listener to be enabled. Topic filter is one subscription pattern devices publish to, such as devices/+/data. QoS controls MQTT delivery guarantee from device to gateway. Payload JSON is parsed into fields; raw stores the original payload. Device ID source maps messages using a topic segment, payload field, client ID, or fixed value. Pipeline stores first, runs anomaly detection, then forwards only when a Data Forwarding profile is selected.
